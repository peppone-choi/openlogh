<!-- Reverse-engineered MDX binary format from investigation -->

VB Descriptor layout (40 bytes, found via marker scan at +0x18):

```
+0x00: u32 type        (0=stride-72, 1=stride-84)
+0x04: u32 ptr_vb      (runtime address — not a file offset)
+0x08: u32 vertex_count (GROUND TRUTH vertex count)
+0x0C: u32 ptr_ib      (runtime address — not a file offset)
+0x10: u32 index_count  (GROUND TRUTH index count — KEY FIX)
+0x14: u32 ptr_???     (runtime address)
+0x18: u32 marker      (74002=stride-72, 336402=stride-84)
+0x1C: u32 padding     (0)
+0x20: u32 stride      (72 or 84)
+0x24: u32 type2       (1 or 2 — possibly strip vs list flag)
```

Vertex layouts:
stride-72: pos@0(3f), nrm@12(3f), uv@24(2f), W=1.0@32
stride-84: W=1.0@0, nrm@16(3f), pos@52(3f), uv@68(2f)

File layout (per mesh group, e.g., body_a / body_b):
[Header region: TOC + scene graph + materials + descriptors]
[VB0 data: vertex_count_0 * stride_0 bytes]
[IB0 data: index_count_0 * 2 bytes — indices reference combined VB pool]
[VB1 data: vertex_count_1 * stride_1 bytes]
[IB1 data: index_count_1 * 2 bytes — indices reference combined VB pool]
...repeat...

Cross-VB indexing:

- Indices in IB0 that are >= vertex_count_0 reference vertices in VB1
  with offset = index - vertex_count_0
- This means all VBs under one descriptor group form a combined pool
- The descriptor ic field gives EXACT index count (no need for heuristic scan)

Key measurements (Brunhild example):
Descriptor 0: vc=2943, ic=9297, stride=72, marker=74002
Descriptor 1: vc=3104, ic=7191, stride=84, marker=336402
VB[0]: 0x4430-0x37fe8 (2943*72), IB[0]: 0x37fe8 + 9297*2
After IB[0]: more u16 indices (cross-VB, values 2943-3284)
VB[1]: 0x3dfa8-0x7da28 (3104\*84), IB[1]: 0x7da28 + 9674 indices
VBs 2-3: smaller parts (62v stride-72, 307v stride-84)

  <files>backend/scripts/mdx_scene_graph_parser.py</files>
  <action>
Create `backend/scripts/mdx_scene_graph_parser.py` that implements descriptor-driven extraction:

**Step 1 — Descriptor discovery:**

- Scan entire file for 40-byte VB descriptors by matching marker (74002 or 336402) at +0x18 AND stride (72 or 84) at +0x20
- Validate: vertex_count in [3, 100000], index_count in [3, 500000]
- Deduplicate overlapping descriptors (within 40 bytes of each other)
- Store type, vertex_count, index_count, marker, stride, type2

**Step 2 — VB location via W=1.0 heuristic (reuse from mdx_extract_v2.py):**

- For each descriptor, find the W=1.0 run matching its vertex_count
- stride-72: W=1.0 at offset 32 within each vertex
- stride-84: W=1.0 at offset 0 within each vertex
- Match descriptor to W run by vertex_count equality (or >= with fallback)
- If no W run found, try descriptor offset + 40 as VB start (with validity check)

**Step 3 — IB reading using descriptor ic (THE KEY FIX):**

- IB starts immediately after VB end (vb_start + vertex_count \* stride)
- Read EXACTLY index_count u16 values from IB start — do NOT stop early on "out of range" values
- This is the critical difference from existing extractors which break on index >= vc
- However: if a descriptor's ic seems wrong (indices produce degenerate geometry), fall back to contiguous-valid-index scan as existing code does

**Step 4 — After-descriptor IB discovery (for cross-VB indices not covered by descriptors):**

- After reading the descriptor-defined IB, scan forward for additional u16 runs that reference the combined VB pool
- Combined VB pool: all VBs sorted by file offset, vertices numbered sequentially (VB0 starts at 0, VB1 starts at VB0.vc, etc.)
- Additional index runs (between IB end and next VB start) use this combined numbering
- Read these as separate draw calls / sub-meshes

**Step 5 — Face decoding with strip/list auto-detection:**

- Reuse the decode_best() approach from mdx_extract_final.py
- Try triangle strip (degenerate = restart) AND triangle list, pick by avg aspect ratio
- Also check descriptor type2 field (1 vs 2) as a possible strip/list indicator
- Fix winding using vertex normals (reuse fix_winding from mdx_extract_final.py)
- Filter degenerate faces (zero area) and thin faces (aspect > 80)

**Step 6 — OBJ output:**

- Each MDX file produces one OBJ with all mesh parts merged
- Write v/vn/vt/f with proper 1-based indexing and vertex base offsets per part
- Support --out directory, --no-mirror, --verbose flags
- X-axis mirror for symmetric ships (reuse mirror logic from mdx_extract_v2.py)
- Report per-file vertex/face counts and overall stats

**Step 7 — Combined VB pool index resolution:**

- When an index value >= current VB's vertex_count, resolve it against the combined pool
- combined_pool[i] = (vb_index, local_vertex_index) mapping
- Read vertex data from the correct VB using the resolved local index
- This handles the cross-VB indices that current extractors miss

**CLI interface:**

```
python mdx_scene_graph_parser.py <mdx_file_or_dir> [--out DIR] [--no-mirror] [--verbose] [--info]
```

- Directory mode: process all \*\_h.mdx files
- --info: print descriptor/VB/IB info without writing OBJ
- --verbose: show detailed per-VB/IB diagnostics
- Exit code 0 if all files processed successfully

**Important anti-patterns to avoid:**

- Do NOT use local vertex_count as max valid index for IB reading — use descriptor ic for exact count
- Do NOT assume IB has a gap before it — IB starts immediately at VB end
- Do NOT assume all indices reference only the local VB — cross-VB indices are valid
- Do NOT skip indices > local vc — they reference the combined VB pool
