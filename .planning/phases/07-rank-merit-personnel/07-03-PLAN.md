# Plan 07-03: Frontend Rank Display + Personnel Panel

## Objective
Add TypeScript types, API client, Zustand store, and UI components for rank display and personnel management.

## Tasks

### Task 1: TypeScript Types
type="auto"

Add rank/personnel types to frontend types:
- `RankLadderEntry`: matches backend DTO
- `PersonnelInfo`: matches backend DTO
- `PromoteDemoteRequest`, `PersonnelActionResponse`

**Done criteria:** Types added to types/index.ts or types/personnel.ts.

### Task 2: Personnel API Client
type="auto"

Add personnel API functions to frontend lib:
- `fetchRankLadder(sessionId, factionId)`
- `fetchPersonnelInfo(sessionId, officerId)`
- `promoteOfficer(sessionId, request)`
- `demoteOfficer(sessionId, request)`

**Done criteria:** API functions exist in lib/api/ directory.

### Task 3: Personnel Store
type="auto"

Create Zustand store for personnel state:
- `rankLadder`, `personnelInfo`, `loading`, `error`
- Actions: `loadLadder`, `loadInfo`, `promote`, `demote`

**Done criteria:** personnelStore.ts exists with all state and actions.

### Task 4: Rank Display Component
type="auto"

Create RankBadge component showing rank tier, title (faction-aware), and merit progress bar. Reusable across officer info panels.

**Done criteria:** RankBadge component renders rank info.

### Task 5: Personnel Panel Component
type="auto"

Create PersonnelPanel component:
- Shows rank ladder table for selected faction
- Shows personnel info for selected officer
- Promote/demote buttons visible only for officers with authority (position card check)
- Merit progress visualization

**Done criteria:** PersonnelPanel component exists with ladder view and action buttons.

### Task 6: Build Verification
type="auto"

Run frontend build to verify no TypeScript errors.

**Done criteria:** `pnpm build` succeeds or `pnpm tsc --noEmit` passes.

## Verification
- All TypeScript types match backend DTOs
- API client calls correct endpoints
- Store manages rank/personnel state
- Components render without errors
