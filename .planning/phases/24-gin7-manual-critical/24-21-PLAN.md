# Phase 24-21 — gin7 자동 생산 품목 시드 데이터 검증

## 1. 배경

gin7 매뉴얼 부록 p75-77 (원문 표제: "自動生産品目 一覧") 은 제국·동맹·페잔
영역의 각 성계/행성에서 **자동 생산되는 함종**을 테이블로 명시한다.
한 성계가 한 턴마다 자동으로 생산하는 함선이 battleship / cruiser /
destroyer / carrier / transport / hospital / fortress 중 어느 타입인지가
게임 초기 밸런스와 직결된다.

OpenLOGH 현 상태:
- `backend/game-app/src/main/resources/ship_stats_*.json` — 함종별 베이스
  스펙 (가격/훈련도/탑재량) 은 존재.
- `docs/scenarios.json` — 시나리오별 진영·자본 매핑은 존재.
- **그러나** 성계/행성별 자동 생산 함종 테이블은 어디에도 없다.
  `PlanetProductionService` 류의 서비스도 존재하지 않는다.

결과: 실가동 세션에서 모든 행성이 똑같이 mixed 생산을 해서 gin7 의
"특정 성계는 순양함만, 특정 행성은 수송함만" 같은 산업 특화가 재현되지
않는다. 매뉴얼 §7.1 의 약 60+ 성계/행성 데이터가 미반영.

## 2. 스코프 (Plan only)

본 phase 는 **Plan 문서만** 작성하고 실제 데이터/코드 변경은 후속
phase 로 분리한다. 이유는:

1. 매뉴얼 p75-77 3페이지에 약 **60+ 레코드** 가 나열되어 있고, 각
   레코드는 성계명·혹성명·생산함종 3 컬럼이므로 체계적으로 옮기려면
   전용 탭 하나가 필요하다.
2. 현재 시나리오 seed 는 캐논 LOGH 배치를 따르지 않는 영역도 있어서,
   매뉴얼 데이터를 그대로 import 하기 전에 시나리오별 "LOGH 원작 정렬"
   정책부터 확정해야 한다 (삼국지 → LOGH 리테마 매핑).
3. `Planet` 엔티티에 `autoProductionShipClass` 필드가 없으므로 Flyway
   마이그레이션이 선행되어야 한다 — 코드 변경과 데이터 변경이 최소 2
   PR 로 쪼개져야 안전하다.

## 3. 후속 phase 분기

### 24-21a — 인프라 선행 (코드)
- `Planet.autoProductionShipClass: String?` 필드 추가 (nullable, null =
  auto production 없음).
- Flyway V55: `alter table planet add column auto_production_ship_class
  varchar(32)` + default null.
- `PlanetProductionService` 신규 — 매 턴마다 `autoProductionShipClass`
  가 non-null 인 행성에서 1 유닛씩 해당 함종 건조.
- 테스트: 생산 로직 단위 테스트 + Flyway 마이그레이션 smoke.

### 24-21b — 시드 변환 (데이터, pure)
- 매뉴얼 p75-77 을 csv / jsonl 로 1회 필사 →
  `backend/game-app/src/main/resources/data/gin7_auto_production.csv`.
- 필드: `starSystemName, planetName, shipClass, factionType, notes`.
- OpenLOGH 의 현 scenarios.json 과 매뉴얼의 성계/행성 네임스페이스가
  완전히 일치하지 않으므로 name normalizer (trim + 삼국지 레거시
  별칭 제거) 가 선행되어야 한다.

### 24-21c — 연결 (마이그레이션 + 검증)
- Scenario loader 가 24-21b 의 csv 를 읽어 해당 Planet row 의
  `autoProductionShipClass` 를 set. 신규 시나리오 로딩 시와 기존 세션
  리마이그 시 모두 동작해야 한다.
- `gin7-manual-complete-gap.analysis.md §7.1` 의 매치율을 0% → ~100%
  로 갱신하는 검증 리포트.
- Zero-script QA: 특정 성계의 1달치 턴 진행 후 함선 개수/함종이 매뉴얼
  기대값과 일치하는지 확인.

## 4. 리스크 / 사전 의사결정 필요 항목

1. **네이밍 매핑**: 매뉴얼은 은하영웅전설 원작의 성계 (오딘, 바랄랏,
   캅헤, 이제르론, 하이네센, 페잔 등) 을 사용하지만 현 프로젝트의
   scenarios.json 은 일부 삼국지 행성명을 그대로 쓴다. → 변환표가
   선행되어야 함.
2. **Production 스케일**: 매뉴얼은 "1 턴 = 1 유닛 자동 생산" 을 명시
   하지만 OpenLOGH 의 tick/turn 주기는 다르다. 스케일 계수 확정 필요.
3. **시나리오 분리**: 전체 시나리오 대 캐논 LOGH 시나리오 각각에
   시드가 다를 수 있음. 시나리오별 파일로 분리할지 단일 테이블을 둘지
   결정 필요.
