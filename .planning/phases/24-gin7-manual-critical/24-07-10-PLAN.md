# Phase 24-07/08/09/10 — Large Tactical & System Gaps (Combined Plan Sketch)

These four plans each need a full session of dedicated design once picked up, so the
sketches below stay short. They exist here to (1) prove the gap analysis findings have
been understood at the right level of depth and (2) provide enough scaffolding that a
future session can promote one sketch to its own `24-NN-PLAN.md` file without re-doing
the reconnaissance work.

---

## 24-07 — 戦死 → 帰還惑星 워프 (gap A7 / C4)

### Manual reference

gin7 manual p51 (Chapter 4, 負傷 処理): 旗艦が破壊されたキャラクターは"負傷"状態となり、
設定された帰還惑星へ瞬時ワープする。長官が戦闘中にいなくなる場合の処理。

### Current state (post-reconnaissance)

- **`DeathInjurySystem.kt` does not exist.** (gap analysis document was wrong.) Only
  `InjuryEvent` — and that is a stat-attrition event used by
  `Gin7EconomyService.processDisasterOrBoom`, not a combat death signal.
- `TacticalBattleEngine:398` does emit some kind of InjuryEvent but on inspection it is
  for 지휘력 저하 during a regiment's commander being wounded, not 旗艦 파괴.
- Officer has no `returnPlanetId` field. It does have `planetId` (current location) and
  there is no concept of a home planet.

### Proposed design

1. **New field** `Officer.returnPlanetId: Long = 0L` defaulting to `planetId` at
   creation. Flyway migration.
2. **New system** `engine/tactical/DeathInjurySystem.kt`:
   - `handleFlagshipDestroyed(officer: Officer, battle: TacticalBattleState)` → set
     `officer.injury = 1`, warp officer to `returnPlanetId`, emit `OfficerReturnedEvent`
     via GameEventService, remove officer from `battle.participatingOfficers`
3. **Wire** into `TacticalBattleEngine` flagship-HP-zero branch.
4. **Admin command** 帰還惑星設定 (manual p36 implies a per-officer setting UI) — the
   simplest first pass is a PERSONAL command `SetReturnPlanetCommand` keyed off the
   officer's current location.

### Effort

2 sessions (design + implement + test + frontend admin card).

## 24-08 — 함정 射線判定 (gap C1)

### Manual reference

gin7 manual p49 (Chapter 4, 戦闘システム): ビーム・ガン・ミサイル攻撃は、射線上に
味方ユニットまたは射線障害地形 (プラズマ嵐・サルガッソ) があると遮断される。
要塞砲は既に射線判定あり。艦vs艦の friendly-fire / 地形遮断は未実装。

### Current state

- `engine/tactical/weapons/BeamWeaponSystem.kt`, `GunWeaponSystem.kt`,
  `MissileWeaponSystem.kt` compute damage as a distance-only attenuation.
- Fortress cannon has line-of-sight, so the pattern already exists for reuse.

### Proposed design

1. **New utility** `engine/tactical/util/LineOfSight.kt`:
   - `isBlocked(fromX, fromY, toX, toY, units: List<TacticalUnit>, terrain: Terrain): Boolean`
   - Bresenham-style ray traversal across a coarse grid (1 cell = 1000 LY, adjustable)
2. **Modify** each of the three weapon systems to call `isBlocked` before applying
   damage; blocked shots are logged as "味方誤射を回避" but still consume ammo.
3. **Terrain field** — `TacticalBattleState` already has some terrain concept;
   extend with `plasmaStorms: List<PlasmaStorm>` and `sargassos: List<Sargasso>` if not
   present. Verify in reconnaissance pass.

### Effort

3 sessions. Weapon balance impact is large — every tactical engine replay test will
need re-baselining.

## 24-09 — 叙勲 Medal System (gap A3)

### Manual reference

gin7 manual p35 (Chapter 3, 人事/叙勲) + p58 appendix (階級ラダー 제3法則 — 동계급자간
정렬: 勲章 수 > 功積 > 在職年数).

### Current state

- `AwardDecorationCommand.kt` exists and is registered as "서훈" but its body is a stub
  that only sets `destOfficer.meta["awarded"] = true`.
- Officer has no `medals` field. `Medal` entity does not exist.
- `RankLadderService.kt` sorts officers by `officerLevel` only; the tiebreaker on medal
  count is missing, so two commodores with different medal counts currently sort by
  whatever secondary comparator the list order happens to have.

### Proposed design

1. **New entity** `entity/Medal.kt` — id, session_id, officer_id, code (enum),
   awarded_at, awarded_by (officer_id), citation (text).
2. **New field** `Officer.medalCount: Int = 0` denormalized cache for sort performance.
3. **Modify** `AwardDecorationCommand`:
   - Insert Medal row, increment `destOfficer.medalCount`, push global history log
4. **Modify** `RankLadderService.sortOfficers`:
   - Primary: officerLevel desc
   - Secondary: medalCount desc (NEW)
   - Tertiary: experience desc (existing)
   - Quaternary: createdAt asc (tiebreaker)
5. **New constraint** `MedalAuthorityConstraint` — only ranks 8+ (대장 이상) can award
   medals per manual p35.

### Effort

2 sessions.

## 24-10 — 叛乱 → 反乱軍 진영 분리 (gap E51)

### Manual reference

gin7 manual p40 (Chapter 3, 反乱). 「謀反」커맨드 성공 후 해당 장교 + 그 휘하 함대가
새 진영 "반란군" 으로 분리된다. 반란군은 AI 진영으로 취급되며 원래 진영과 자동 선전포고
상태가 된다.

### Current state

- `RebellionCommand.kt` (in `PoliticalPersonalCommands.kt`) exists but its body is one
  line: `general.meta["rebellionStatus"] = "ACTIVE"`. No faction is created, no fleets
  are moved, no diplomacy state is set.
- `SessionState` already models `factions: List<Faction>` so a new faction can be
  created without schema changes.
- `Faction.factionType` has a `rebel` code defined in CLAUDE.md but no instances exist.

### Proposed design

1. **Modify** `RebellionCommand.run()`:
   - Create new `Faction(factionType = "rebel", ...)` via `factionRepository.save`
   - Move `general.factionId` to the new faction
   - Move all fleets where `fleet.commanderOfficerId == general.id` into the new faction
   - Move all position cards that represent chain-of-command (from the participating
     commanders only; rank-and-file stay with the old faction)
   - Create `Diplomacy(srcFactionId = newRebel, destFactionId = original, stateCode = "전쟁")`
   - Emit `RebellionSucceededEvent` via GameEventService
2. **New constraint** `RebellionAuthorityConstraint`:
   - Minimum `officerLevel` = 5 (준장 이상) per manual hints
   - Forbid if already rebelling or if in a rebel faction
3. **Integration test** in `ScenarioPlayableIntegrationTest` — scripted rebellion
   scenario that ends with two factions and an active war diplomacy row.

### Effort

2 sessions. Cross-cutting concerns (event bus, diplomacy service, fleet ownership) make
this larger than it looks.
