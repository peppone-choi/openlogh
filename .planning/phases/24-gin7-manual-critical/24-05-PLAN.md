# Phase 24-05 — Full CP Rebalance (commands.json runtime binding)

## Problem statement

`BaseCommand.getCommandPointCost()` returns a hard-coded `1` by default and is not
overridden by any of the 81 gin7 command subclasses (except the three spot-fixes landed
in Plan 24-01-04). `RealtimeService.scheduleCommand()` line 317 calls
`command.getCommandPointCost().coerceAtLeast(1)` and passes the result to
`cpService.deductCp()`. Therefore every strategic command, regardless of manual value,
currently charges exactly 1 CP at runtime.

`backend/shared/src/main/resources/data/commands.json` already contains the authoritative
manual values (spot-checked: `governance_goal.cpCost = 80`, `arrest_authorization.cpCost
= 800`, `execution_order.cpCost = 800`, etc.) and `CommandRegistry.lookupById()` is
already in place — but that lookup path is dead code. `CommandRegistry.load()` has zero
runtime callers outside of `CpCostBalanceTest`.

This makes CP balance almost entirely cosmetic today, which is why the manual-prescribed
4x and 5x delta on 通治目標 / 逮捕許可 / 執行命令 never surfaced as a gameplay complaint —
nothing was actually charging the CP.

## Goal

Make `BaseCommand.getCommandPointCost()` read from `commands.json` at runtime, keyed on
the command's registered Korean action name, so that all 81 commands automatically pick
up the manifest values. Keep the per-command override path as an escape hatch for
commands whose CP depends on runtime state (e.g. warp distance scaling — see manual p38).

## Design options

### Option A — Pass a CpCostRegistry into OfficerCommand constructor

Pros: explicit dependency, test-friendly via constructor injection.
Cons: touches every one of the ~85 `class FooCommand(general, env, arg)` constructors in
`command/gin7/**/*.kt`. High merge-conflict risk.
Verdict: rejected — too much surface area.

### Option B — Singleton object initialized lazily from classpath JSON

Pros: no constructor changes. `BaseCommand` can call
`CommandCpCostRegistry.get(actionName)` directly. Works in tests because
`commands.json` is already on the test classpath. Easy rollback if JSON format changes.
Cons: global state. Must be careful with hot-reload in Spring dev mode. Need a fallback
when the action name is missing from JSON (presumably return 1 and log a warning).
Verdict: **recommended**.

### Option C — Inject via Spring @Component and @Autowired into BaseCommand setter

Pros: Spring-idiomatic, scoped.
Cons: `BaseCommand` is instantiated by `commandRegistry.createOfficerCommand()` lambdas
outside of Spring, so setter injection would need manual plumbing at every lambda. High
boilerplate for zero benefit over Option B.
Verdict: rejected.

## Implementation sketch (Option B)

1. **New file** `backend/game-app/src/main/kotlin/com/openlogh/command/CommandCpCostRegistry.kt`:

   ```kotlin
   object CommandCpCostRegistry {
       private val byActionName: Map<String, Int> by lazy {
           val registry = com.openlogh.shared.model.CommandRegistry.load()
           registry.getAllCommands().associate { it.nameKo to it.cpCost }
       }
       fun get(actionName: String): Int = byActionName[actionName] ?: DEFAULT
       private const val DEFAULT = 1
   }
   ```

2. **Modify** `backend/game-app/src/main/kotlin/com/openlogh/command/BaseCommand.kt`
   line 43:

   ```kotlin
   open fun getCommandPointCost(): Int = CommandCpCostRegistry.get(actionName)
   ```

3. **Revert** the three spot-fix overrides added in 24-01-04 (or leave them — they become
   redundant no-ops once the lookup works). Leaving them is probably safer for the first
   landing because they serve as regression anchors.

4. **Extend** `CpCostBalanceTest.kt` with a new test:

   ```kotlin
   @Test
   fun `every registered command cost matches commands json`() {
       val gin7 = Gin7CommandRegistry()
       gin7.getGeneralCommandNames().forEach { name ->
           val cmd = gin7.createOfficerCommand(name, dummyOfficer(), dummyEnv(), null)
           val expected = CommandCpCostRegistry.get(name)
           assertEquals(expected, cmd.getCommandPointCost(),
               "command $name getCommandPointCost mismatch with commands.json")
       }
   }
   ```

5. **Update** `.planning/STATE.md` with the CP-system rebinding note so that future
   contributors don't re-introduce spot-fixes instead of reaching for the lookup.

## Risk

- **Behavior change is large.** Every 4-CP command now suddenly costs the manual value
  (often 80, 160, 320). Officer `pcpMax`/`mcpMax` defaults to 5 (Officer.kt line 247-250),
  which means **every command becomes instantly unexecutable** under the current pool
  sizing. Before landing 24-05 we MUST also resize the pool defaults — or the whole game
  breaks for every session started after the migration.

  Proposed pool resize (match manual p33): PCP base = 800, MCP base = 800, both scaling
  with the officer's rank via `officerLevel * 80`. This is a separate migration and
  should be its own commit (24-05a Pool-Size Migration) landing **before** 24-05 itself.

- **Hot-reload in dev.** `CommandCpCostRegistry.byActionName` uses `lazy` which caches
  the first read. If `commands.json` is edited during a running dev server the cache
  won't refresh. For now, document this as a known limitation — a watcher-based reload
  can come later if it becomes painful.

## Out of scope for 24-05

- Warp distance–scaled CP for 워프항행 (see `WarpNavigationCommand.kt:13` — `cpCost=40`
  is a placeholder "Phase 3에서 거리 연동 예정"). This stays a per-command override.
- Modifier-based CP discounts (personality, special abilities). Phase 25-high track.
