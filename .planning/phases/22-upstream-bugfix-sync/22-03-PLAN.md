# Plan 22-03: EconomyService — per-resource schedule split

## Objective

Port the upstream EconomyService schedule fix from commit `a7a19cc3`. Income/salary/upkeep was being called from `preUpdateMonthly` + `postUpdateMonthly` on every turn, causing 12x annual drain. Upstream moved it to event-driven per-resource processing scheduled for month 1 (gold/funds) and month 7 (rice/supplies) only.

## LOGH architectural difference

LOGH currently has two economy services:
- `EconomyService.kt` — legacy, with `processIncomeEvent` / `processSemiAnnualEvent` currently as STUBS pointing to "TODO Phase 4: gin7EconomyService.processIncomeEvent"
- `Gin7EconomyService.kt` — new gin7-compliant service, partially implemented

This plan ports the upstream fix pattern to the LOGH architecture:

**If Gin7EconomyService is the intended future-state:** implement the per-resource split there and wire the stubs in EconomyService to route accordingly (current Phase 4 TODO).

**If EconomyService legacy path is still active in turn pipeline:** port the full upstream fix to EconomyService (preUpdateMonthly no-op the income calls, split processIncome/processSemiAnnual by resourceType).

Check the turn pipeline (`TickEngine` / `TurnService` / `EconomyPreUpdateStep` / `EconomyPostUpdateStep`) to determine which service is actually called each tick — that determines the porting target.

## Upstream fix summary (for reference)
