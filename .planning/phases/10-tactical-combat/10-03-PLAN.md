# Plan 10-03: WebSocket Battle Controller & REST API

## Objective
Wire the tactical battle engine to WebSocket for real-time battle updates and provide REST endpoints for battle management (energy allocation, formation changes, battle queries).

## Context
- @backend/game-app/src/main/kotlin/com/openlogh/config/WebSocketConfig.kt — existing WebSocket config
- @backend/game-app/src/main/kotlin/com/openlogh/service/GameEventService.kt — existing broadcast methods
- @backend/game-app/src/main/kotlin/com/openlogh/engine/tactical/ — new tactical engine (Plan 10-02)

## Tasks

### Task 1: Create TacticalBattleService and WebSocket controller
type="auto"

Create:
1. `com.openlogh.service.TacticalBattleService` (@Service):
   - `startBattle(sessionId, starSystemId, attackerFleetIds, defenderFleetIds): TacticalBattle`
   - `processBattleTick(battleId: Long)` — advance one tick, broadcast state via WebSocket
   - `setEnergyAllocation(battleId, officerId, allocation: EnergyAllocation)` — player command
   - `setFormation(battleId, officerId, formation: Formation)` — player command
   - `retreat(battleId, officerId)` — attempt retreat (requires WARP energy)
   - `endBattle(battleId)` — finalize battle, persist results, update fleet HP/ships
   - Active battles stored in-memory ConcurrentHashMap for tick processing
2. `com.openlogh.controller.TacticalBattleController` (@Controller for WebSocket):
   - `@MessageMapping("/battle/{sessionId}/command")` — receive player commands (energy, formation, retreat)
   - Sends to `/topic/world/{sessionId}/tactical-battle/{battleId}` — per-battle state updates
3. `com.openlogh.controller.TacticalBattleRestController` (@RestController):
   - `GET /api/v1/battle/{sessionId}/active` — list active battles
   - `GET /api/v1/battle/{sessionId}/{battleId}` — get battle state
   - `GET /api/v1/battle/{sessionId}/{battleId}/history` — get battle tick history
4. DTO classes in `com.openlogh.dto.TacticalBattleDtos`

Done when: WebSocket controller compiles, REST endpoints defined, service manages battle lifecycle.

### Task 2: Integrate battle tick processing with game tick engine
type="auto"

Wire tactical battles into the existing tick loop:
1. Add `TacticalBattleService` to tick engine so active battles process each server tick
2. Battle state broadcast happens after each tick via `GameEventService.broadcastBattle()`
3. Battle completion triggers fleet HP/ship updates back to Fleet entities
4. Battle result fires `BattleEvent` through existing `GameEventService`

Done when: Tactical battles tick alongside the game engine, results persist to DB.

## Verification
- All new Kotlin files compile: `cd backend && ./gradlew :game-app:compileKotlin`

## Success Criteria
- Players can send energy/formation commands via WebSocket during battle
- Battle state broadcasts to all participants each tick
- REST API returns active battles and battle state
- Battle results update fleet entities when battle ends
