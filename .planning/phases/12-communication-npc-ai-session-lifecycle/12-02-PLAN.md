# Phase 12 Plan 02: Victory Conditions Service

## Objective
Implement VictoryService that checks 3 victory conditions every tick, produces 4-tier evaluation, and triggers session end.

## Context
- @docs/reference/scenarios_detail.md — victory conditions reference
- @CLAUDE.md — Victory Conditions section: capital capture, 3-system threshold, UC801.7.27 time limit
- @backend/game-app/src/main/kotlin/com/openlogh/entity/SessionState.kt — session state entity

## Tasks

### Task 1: VictoryCondition model and VictoryService
type="auto"

**Do:**
1. Create `VictoryCondition` enum: `CAPITAL_CAPTURE`, `SYSTEM_THRESHOLD`, `TIME_LIMIT`
2. Create `VictoryTier` enum: `DECISIVE`, `LIMITED`, `LOCAL`, `DEFEAT` with Korean labels (결정적/한정적/국지적/패배)
3. Create `VictoryResult` data class: `condition`, `tier`, `winnerFactionId`, `loserFactionId`, `stats` map
4. Create `VictoryService` in `com.openlogh.service` with:
   - `checkVictoryConditions(sessionId: Long): VictoryResult?` — checks all 3 conditions
   - `checkCapitalCapture(sessionId)`: enemy faction's capital planet owned by another faction
   - `checkSystemThreshold(sessionId)`: enemy faction holds 3 or fewer star systems (including capital)
   - `checkTimeLimit(sessionId)`: current game date >= UC801.7.27, compare total population
5. Tier calculation:
   - DECISIVE: capital captured OR enemy reduced to 0 systems
   - LIMITED: enemy at 1-3 systems
   - LOCAL: time limit with > 2x population advantage
   - DEFEAT: time limit with <= 2x advantage (relative to the weaker side)
6. Create Flyway V39 adding `victory_result` JSONB column to `session_state` table

**Done when:**
- VictoryService checks all 3 conditions
- Tier evaluation produces correct VictoryTier
- V39 migration adds victory_result column

### Task 2: Victory check integration with tick engine
type="auto"

**Do:**
1. Create `VictoryCheckTask` that runs every 60 ticks (1 minute real-time) within the tick engine
2. When victory detected:
   - Set session status to 'ENDED'
   - Store VictoryResult in session_state.victory_result
   - Broadcast victory event via WebSocket to all connected clients
   - Log victory event to Event table
3. Create `VictoryController` REST endpoint:
   - `GET /api/v1/world/{sessionId}/victory` — returns victory result if session ended
4. Create DTOs: `VictoryResultDto`, `VictoryEventDto`

**Done when:**
- Victory check runs periodically in tick engine
- Victory triggers session end and WebSocket broadcast
- REST API exposes victory result

## Verification
- VictoryService compiles and all 3 conditions are implemented
- VictoryController endpoint exists
- VictoryTier enum has 4 values with Korean labels

## Success Criteria
- Capital capture ends the game immediately
- Reducing enemy to 3 or fewer systems ends the game
- Time limit (UC801.7.27) triggers population comparison
- Session end produces correct 4-tier evaluation
