<!-- Each call site must use the Kotlin equivalent that matches the PHP function used in legacy -->

PHP (int)$x          --> Kotlin .toInt()              EXACT MATCH (both truncate toward zero)
PHP round($x)        --> Kotlin kotlin.math.round(x).toInt()   CLOSE (diverges at .5 only)
PHP intdiv($a, $b)   --> Kotlin a / b                 EXACT MATCH (same-sign operands)
PHP floor($x)        --> Kotlin floor(x).toInt()      EXACT MATCH
PHP ceil($x)         --> Kotlin ceil(x).toInt()        EXACT MATCH

Divergence summary:
  Math.round(2.5).toInt()   = 3   (Java Long-returning, half-up)
  (2.5).roundToInt()        = 2   (Kotlin banker's rounding, half-to-even)
  kotlin.math.round(2.5).toInt() = 2  (same as roundToInt, returns Double)
  PHP round(2.5)            = 3   (half-away-from-zero)

For PHP round() parity at .5 boundaries:
  - Math.round(x).toInt() matches PHP round(x) for POSITIVE .5 values
  - Math.round(x).toInt() DIFFERS from PHP round(x) for NEGATIVE .5 values
  - In practice: exact .5 values are rare in game formulas (random * multiplier rarely hits .5)
  - Focus on formulas with deterministic .5 outputs: value/2, value*0.5, value*1.5

    Per D-06 (audit and normalize inconsistent rounding) and D-07 (priority order: EconomyService > GeneralAI > BattleEngine > commands):

    **Audit method:** For each file, search for every `.roundToInt()` and `Math.round(` call. For each occurrence:
    1. Determine which legacy PHP function the formula corresponds to:
       - If the legacy PHP uses `(int)` cast (truncation) -> the Kotlin site should use `.toInt()`, NOT `.roundToInt()`
       - If the legacy PHP uses `round()` (half-away-from-zero) -> the Kotlin site should use `Math.round(x).toInt()` for positive-only values, or keep `.roundToInt()` if exact .5 values are impossible for that formula
       - If the legacy PHP uses `floor()` -> keep `floor().toInt()`
       - If the legacy PHP uses `ceil()` -> keep `ceil().toInt()`
    2. If the current Kotlin method does not match the PHP equivalent, change it

    **Primary normalization: Replace `Math.round(x).toInt()` with `kotlin.math.round(x).toInt()`**
    Per RESEARCH pitfall #1: `Math.round(Double)` returns Long, and `.toInt()` hides a Long-to-Int narrowing. Replace all 13 `Math.round(x).toInt()` occurrences with `kotlin.math.round(x).toInt()` which returns Double directly. For the game's value ranges (< 100,000) this is safe and eliminates the narrowing risk. Add `import kotlin.math.round` where needed.

    **Secondary normalization: Audit `.roundToInt()` sites**
    For each of the ~90 `.roundToInt()` call sites:
    - If the formula is equivalent to PHP `(int)` cast (truncation toward zero), replace `.roundToInt()` with `.toInt()`
    - If the formula is equivalent to PHP `round()`, keep `.roundToInt()` (for most game values the .5 divergence is negligible since exact .5 outputs are rare with random multipliers)
    - If the formula explicitly divides by 2 or multiplies by 0.5 (deterministic .5 output), and the PHP source uses `round()`, note it in a comment: `// PHP round() parity: .5 divergence possible but matches for non-.5 inputs`
